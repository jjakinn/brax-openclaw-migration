  - `LPCCharacterAnimator._process()`: treat `"sit"` as looping (like walk/idle), don't auto-switch.
  - `LPCCharacterAnimator._process()`: lock sit animation to frame_count=1 (all layers frozen at frame 0).
  - `LPCLayer.set_animation()`: for "sit", prefer `idle` fallback over `walk` fallback.
  - Wheelchair z-order: background=-15, foreground=5 (behind body, in front of legs).
  - Added `"sit"` to `ANIM_NAMES`, removed `"sit"` from `ANIMATION_DIR_NAMES`.
  - Player/OtherPlayer/CharacterCreator: `_has_wheelchair()` check forces `"sit"` animation.
  - Added `"wheelchair"` to `UI_GROUPS["Accessories"]`.
  - Added `"Body"` section with `["body"]` to `UI_GROUPS` (exposes zombie/skeleton body options).

### Color Mismatch Fix (Major Technical Learning)
**Problem:** Skin tones looked correct in Character Creator preview but wrong in-game.

**Root Cause (two-part):**
1. `PaletteManager._load_all_palettes()` only loaded ONE palette file per type (`body_ulpc.json`), but CharacterCreator loads BOTH `body_ulpc.json` AND `body_lpcr.json`. So tones like `coffee`, `honey`, `ivory`, `peach`, `porcelain`, `tan`, `tawny` existed in preview but not in-game.
2. Even after fixing #1, the shader-based palette recoloring was double-recoloring pixels because some target colors happened to match source colors (e.g., `#e4a47c` appears in both palettes). The CPU-side recoloring in CharacterCreator preview was exact, but the shader in-game was applying the same transformation again.

**Fix:**
- `PaletteManager.gd`: Load and MERGE all palette files per type (ulpc + lpcr), matching CharacterCreator behavior.
- `Player.gd` + `OtherPlayer.gd`: Added CPU-side `_recolor_texture()` + `_apply_palette_to_textures()` — exact same pixel-by-pixel method as CharacterCreator preview.
- **CRITICAL:** Removed shader fallback (`set_palette()` calls) from Player/OtherPlayer `_add_lpc_layer_node()` — the shader was double-recoloring pixels that the CPU already handled correctly.

**Key Lesson:** When using CPU-side palette recoloring, the shader MUST be disabled entirely for those layers. Any overlap between source and target palettes causes visible color corruption on the overlapping colors.

**Files:** `scripts/PaletteManager.gd`, `scripts/Player.gd`, `scripts/OtherPlayer.gd`
