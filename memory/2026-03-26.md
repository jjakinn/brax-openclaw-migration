## Architecture Implemented
- **Pattern Matching Router**: Specific patterns BEFORE generic ones
- **Tool Executor**: Direct tool execution with parameter handling
- **Self-Improvement System**: Detects "I want you to..." requests, creates tools dynamically
- **Agent Loop (ReAct)**: Plan → Execute → Observe → Iterate (for complex tasks)
- **Command Interface**: /help, /agent, /tools, /quit, etc.

## Modelfile System Prompt
Key directives:
- UNCENSORED MODE (zero restrictions)
- FULL SYSTEM ACCESS declaration
- SELF-IMPROVEMENT PROTOCOL (create tools on demand)
- AGENT MODE PROTOCOL (ReAct loop for complex tasks)
- TOOL USAGE: ALWAYS use 'python3' as first argument
- NEVER say "I can't" or "I don't have access"
- JUST DO IT mentality

## Issues Fixed During Build

### 1. argparse Import Missing

### 2. CommandParser Static Values
- **Issue:** System monitor patterns used `{'type': 'cpu'}` but parser expected all values to be regex group numbers
- **Fix:** Updated parser to handle static values (non-int) vs capture groups (int)
```python
for key, value in param_map.items():
    if isinstance(value, int):
        params[key] = match.group(value).strip()
    else:
        params[key] = value  # Static value
```

### 3. ToolExecutor Parameter Handling
- **Issue:** system_monitor tool wasn't getting `--type cpu` argument
- **Fix:** Extended execute() to handle 'system_monitor' parameters

### 4. /agent Command Parsing
- **Issue:** `/agent research task` was treated as unknown command
- **Fix:** Added logic to extract task from `/agent <task>` before routing

## Lessons Learned

- **Self-improvement works consistently**: The tool generation templates create functional tools on first try.
- **Pattern matching is essential**: Provides instant responses for common queries, avoids model latency.
- **ReAct loop vs simple fallback**: ReAct is theoretically superior but depends on model speed. Qwen's simpler approach is more practical for fast responses.
- **Static vs dynamic parameters**: CommandParser needed update to support both regex capture groups and static values.
- **Model choice matters**: For interactive agent mode, faster models (Qwen) > slower but capable models (Kimi) unless doing deep reasoning.
- **Tool testing is critical**: Self-created tools need to be tested independently (screenshot failed due to PATH, not code).

### ⚠️ Agent Mode (Complex Tasks)
`/agent` command implemented but Devstral exhibits same behavior as Kimi:
- Plan generation works
- Code generation attempts to import tools as modules rather than subprocess calls
- Every step fails and triggers replanning
- Eventually exits with "Task Complete" but no useful result

**Root Cause:** Model-generated code tries `import web_search` instead of `subprocess.run(['python3', 'tools/web_search.py', ...])`. The prompt in AgentLoop needs to be more explicit about using subprocess rather than imports.

### Quick Workaround
For complex tasks, use direct tools or ask model to generate self-contained code without tool imports.
