### Fix 8: Large-frame weapons hovering above hand

### Fix 9: Crossbow, wand, rod, whip invisible
- **Crossbow:** `_load_textures` scanned for `walk/<dir_name>.png` but file is `walk/crossbow.png`. Added `_scan_for_png()` fallback.
- **Wand:** `_resolve_asset_bases` only checked top-level dir. Wand's `male/` only contains `slash/` subdir. Added `_subdir_has_png()`.
- **Rod/Whip/Crossbow bg:** Empty directions (dir=2/front transparent). Added `_find_first_valid_direction()` to pre-compute valid direction and fall back in `_update_frame()`.
- Files: `scripts/CharacterCreator.gd`, `scripts/LPCLayer.gd`

### Fix 10: Rod/whip showing attack sprites while walking
- First attempt (`layer_hframes != expected`) BROKE all weapons with non-standard frame counts (longsword_alt=13, etc.). **Reverted.**
- Correct fix: `_scan_for_png_recursive()` and `single_frame` handler now skip `ANIMATION_DIR_NAMES` (slash, thrust, spellcast, hurt, etc.).
- This prevents loading attack animations as walk textures while allowing proper single-frame assets.
- Files: `scripts/CharacterCreator.gd`

### Critical Lesson
`_sample_frame_has_content()` with `step = fs/4` (16px sampling for 64×64) misses small weapons. Any pixel-sampling approach for detecting empty frames is fragile. Direction-level fallback (`_find_first_valid_direction`) is safer than per-frame sampling.
